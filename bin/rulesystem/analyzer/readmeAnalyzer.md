# Static Analyzer

## ¿Qué es el Analyzer?

El Static Analyzer analiza el conjunto de reglas sin ejecutarlas. Trabaja sobre el AST para detectar problemas en la lógica del sistema.


## ¿Para qué sirve?

* Detectar errores antes de ejecutar
* Mejorar la calidad del sistema
* Identificar:

  * Conflictos
  * Redundancias
  * Reglas inactivas


## Tipos de análisis

Conflictos:
Varias reglas generan la misma acción.

Redundancias:
Reglas duplicadas o equivalentes.

Reglas potencialmente inactivas:
Reglas que no pueden activarse por falta de hechos.


## Clases del Analyzer

### StaticAnalyzer.java

Responsabilidad:
Coordinar todos los análisis.

Uso:
StaticAnalyzer analyzer = new StaticAnalyzer();
analyzer.analyze(program);

Internamente usa:

* ConflictDetector
* RedundancyDetector
* InactiveRuleDetector

Errores comunes:

* Incluir lógica directamente en esta clase
* No separar responsabilidades


### ConflictDetector.java

Responsabilidad:
Detectar múltiples reglas con la misma acción.

Cómo funciona:

* Agrupa reglas por acción
* Si hay más de una, hay conflicto

Salida típica:
Action fan_on generated by r1, r2

Errores comunes:

* No agrupar correctamente
* No manejar colecciones adecuadamente


### RedundancyDetector.java

Responsabilidad:
Detectar reglas duplicadas.

Cómo funciona:

* Usa rule.getSignature()
* Agrupa reglas iguales

Salida típica:
Redundant rules: r1, r2

Errores comunes:

* No implementar getSignature()
* No usar asString() en condiciones


### InactiveRuleDetector.java

Responsabilidad:
Detectar reglas que no pueden activarse.

Cómo funciona:

* Obtiene los hechos generados
* Extrae hechos requeridos con extractFacts()
* Compara ambos conjuntos

Salida típica:
Potentially inactive rule: r3

Errores comunes:

* No implementar extractFacts()
* Confundir variables con facts
* Asumir que el análisis es exacto (es una aproximación)


## Relación con el AST

El analyzer depende del AST:

* Rule
* Condition
* Action

El AST no depende del analyzer.


## Limitaciones

* No ejecuta reglas
* No usa estado real
* El análisis de inactividad es aproximado


## Principios aplicados

* Single Responsibility Principle
* Bajo acoplamiento
* Uso de polimorfismo
* Diseño modular


## Errores comunes

* Uso excesivo de instanceof
* Manejo incorrecto de Sets
* No detectar duplicados correctamente
* Mezclar lógica con el intérprete


## Resumen

El analyzer permite validar el sistema de reglas antes de su ejecución, detectando problemas estructurales y mejorando la calidad del programa.
